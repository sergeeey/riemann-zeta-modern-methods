# sandbox/ — буфер сырых гипотез

Зона для быстрого захвата гипотез **ДО** полного FL Full-Ladder ритуала.

## Зачем существует
Основная работа проекта строгая: каждый атом → `experiments/<id>/` с полным
Falsification Ladder (controls, baseline, stress-test, caveats, go/no-go).
Это правильно для валидации, но убивает темп, когда нужно просто **закинуть** идею.

`sandbox/` — это буфер перед гейтом. Сюда падает сырьё. Отсюда выживших
переносим в `experiments/`.

## Что это НЕ
- ❌ **НЕ источник истины.** Ничего отсюда нельзя цитировать как `[VERIFIED]`.
- ❌ **НЕ обязателен FL.** Гипотеза может лежать сырой, неполной, даже сомнительной.
- ❌ **НЕ место для накопления.** Каждая гипотеза обязана получить вердикт в `INDEX.md`.

## Жизненный цикл гипотезы
```
сырьё (note.md)
   │
   ▼
Zero-Signal Gate  ← entity? falsifiable predicate? measurable outcome?
   │
   ├─ прошла ───────► promote → git mv в experiments/<id>/ + полный FL
   ├─ валидна,
   │  но не сейчас ──► parked  (остаётся тут, помечена в INDEX)
   └─ не прошла ─────► drop/REFUSE (остаётся как след, помечена в INDEX)
```

## Структура
```
sandbox/
  README.md                  ← этот файл
  INDEX.md                   ← реестр всех гипотез + вердикт (ОБЯЗАТЕЛЕН)
  <YYYYMMDD>-<слаг>/
    note.md                  ← сырая гипотеза, твоими словами
```

## Workflow
1. Кидаешь гипотезу в чат → я создаю `sandbox/<дата>-<слаг>/note.md` (твои слова, без обработки).
2. Прогоняю через **Zero-Signal Gate** (entity ∧ falsifiable predicate ∧ measurable outcome).
3. Записываю вердикт в `INDEX.md`.
4. promote: `git mv sandbox/<дата>-<слаг> experiments/<id>` — история файла сохраняется.

## Правило против распыления
Гипотеза без вердикта в `INDEX.md` = технический долг. Главный риск проекта —
накопление интересного без доведения. `INDEX.md` этого не позволяет: каждая строка
обязана двигаться (`promote`) либо честно умереть (`drop`/`parked`).
